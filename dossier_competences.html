PROJET MARGARET - Automatisation du traitement des emails d'assurance automobile

Période : Octobre 2024 - En cours

Rôle : Architecte logiciel, Analyste fonctionnel, Développeur lead (équipe de 2 personnes)

Contexte & Besoin métier

Entreprise d'assurance recevant des milliers d'emails de particuliers et professionnels nécessitant une intégration manuelle dans leur logiciel métier. Le processus était 100% manuel, géré par une dizaine de personnes. Processus chronophage et coûteux en ressources humaines. Objectif : automatiser le traitement pour réduire les coûts opérationnels liés à la main-d'oeuvre.

Problématiques techniques

  • Identification automatique du dossier de rattachement de chaque email
  • Classification des types d'emails et de pièces jointes (cartes grises, photos, constats, devis, dossiers de sinistres)
  • Traitement fiable et traçable de grands volumes
  • Gestion des cas d'échec de classification avec un arbre de décision complet : en cas d'échec (sous le seuil de confiance), les documents sont isolés dans un dossier et une notification email est envoyée à un humain pour validation et annotation manuelle

Architecture technique mise en place

  • Event Streaming : Kafka avec ZooKeeper pour l'ingestion et le routage des messages. Outil de visualisation personnalisé developpe pour monitorer les flux
  • Workers Python : Architecture modulaire basée sur un héritage de "workers" liés à des tuyaux Kafka, avec un "worker manager" centralisant les logs et la persistance. L'accent a été mis sur les interfaces (Kafka, MinIO, couche de données) pour rester agnostique aux technologies utilisées
  • IA : Modèles LLM open source. Utilisation uniquement de prompting sur les données de test avec un "system prompt" bien structuré, ce qui s'est avéré suffisant sans fine-tuning
  • Connecteurs ERP : Développement de connectéurs vers le logiciel métier existant. Faute d'API ou d'accès base de données, l'interface a été réalisée vià un outil Python (type RPA) simulant des clics et remplissant des champs de manière fiable sur une machine virtuelle
  • API REST : Serveur d'API commun en librairie Python partagee
  • Monitoring : Application PyQt6 pour le suivi temps réel, OpenTelemetry + ClickHouse pour les logs
  • Stockage : MongoDB pour l'archivage Kafka, base relationnelle (ORM) pour les données persistantes, Minio pour les fichiers
  • Tests : Tests d'intégration mis en place pour vérifier le bon démarrage des services (Docker) et la connexion aux modules. Pas de tests unitaires
  • Déploiement : Via Docker Compose
  • Principes : Architecture orientée objet, haute modularité, couplage faible et cohésion forte

Objectifs & KPI vises

  • Traitement de 2 millions d'emails par an
  • Réduction drastique des coûts opérationnels
  • Amélioration de la fiabilité et de la traçabilité

Technologies : Python, Kafka, IA générative (LLM open source), PyQt6, MongoDB, PostgreSQL, Minio, OpenTelemetry, ClickHouse
